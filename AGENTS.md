## Build

The `.xcodeproj` is **gitignored and generated by XcodeGen** — never edit it by hand; edit `project.yml` instead.

```bash
xcodegen generate                                          # regenerate after any project.yml or new-file change
xcodebuild -project LaunchdManage.xcodeproj -scheme LaunchdManage build   # compile to check for errors
```

Adding/removing/renaming a source file requires re-running `xcodegen generate` before it appears in the build.

## Two-target architecture

This is a non-sandboxed macOS 14+ app (`LaunchdManage`) plus an embedded privileged tool (`PrivilegedHelper`).

- **`LaunchdManage/XPC/HelperProtocol.swift` is compiled into BOTH targets** (see `project.yml` — it's listed as a source of `PrivilegedHelper`). It's the XPC contract. Changing it means changing both sides at once. XPC methods can't return values — they use `withReply:` closures with Cocoa-compatible types only.
- The helper is registered/activated via `SMAppService.daemon(...)` (`XPCClient.swift`), not the legacy SMJobBless. Bundle id is `com.steveshi.launchdmanage.helper`.

### Privilege routing (the central pattern)

Operations on **system-domain** services need root; everything else runs as the user. The split is by `domain == "system"`:

- `LaunchctlService.loadService/unloadService` → system domain calls `XPCClient.shared.*SystemService`; user/gui domains call `ShellExecutor.shared.launchctl(...)` directly (modern `bootstrap`/`bootout` first, falling back to legacy `load`/`unload`).
- `PlistParser` writes go through `XPCClient.shared.writePlist` when root is required, otherwise direct file write.
- `JobCategory` encodes the domain model: `directoryURL`, `domainTarget` (`gui/<uid>` vs `system`), and `isSystemProtected` drive where a job lives and whether it needs the helper.

## Layering

`SwiftUI Views → ViewModels (@Observable @MainActor) → Services (singletons) → {filesystem, launchctl, Unified Logging, XPC}`

- ViewModels and most services are `@MainActor final class` with a `static let shared` singleton (`JobDiscoveryService`, `LaunchctlService`, `PlistParser`, `LogReaderService`, `DiagnosticsService`, `FileWatcherService`, `XPCClient`).
- `JobListViewModel` is the app-wide state, injected via `.environment(...)` in `LaunchdManageApp`. It listens for `.launchdDirectoriesDidChange` (posted by `FileWatcherService`) to auto-reload.
- Multi-window: external/unknown plists open in a separate `WindowGroup(id: "job-editor", for: URL.self)` scene; known launchd plists select in the main list instead (see `handleDroppedURLs`).
- The detail editor is dual-mode (visual `FormEditorView` ↔ live XML via the `CodeEditor` SPM package) with bi-directional sync.

## Security — release gate (unresolved)

**🔴 BLOCKER before any signed public release:** `PrivilegedHelper/main.swift` accepts **every** XPC connection (`shouldAcceptNewConnection` returns `true` unconditionally). The helper's Mach service is system-registered, so any local process can drive root operations (`writePlist`, `loadSystemService`) → local privilege escalation. Fix: `newConnection.setCodeSigningRequirement(...)` (macOS 13+) requiring the same-team-signed main app, returning `false` on mismatch. Deferred because a strict requirement rejects connections under the current ad-hoc signing (`CODE_SIGN_IDENTITY: "-"`); apply together with a real Developer ID.

Defense-in-depth already in place: `HelperTool` restricts root file ops to `/Library/LaunchDaemons` and `/Library/LaunchAgents` (`isPathAllowed`). Keep that allowlist when touching the helper.

## Project conventions (enforced)

- **Swift 6.0, strict concurrency.** Respect actor isolation; services are `@MainActor`, cross-actor data is `Sendable`.
- **No hardcoded user-facing strings.** Use `String(localized: "...", comment: "...")`. Localization is `Localizable.xcstrings` (en + zh-Hans, auto-generated by Xcode; `SWIFT_EMIT_LOC_STRINGS = YES`). For zh-Hans, "Steve Shi" → "轩楝".
- **No new dependencies** without need; bundle ids follow `com.steveshi.<name>`.
- Code comments in this repo are in Chinese — match the surrounding style.
- Versioning lives in `project.yml` (`MARKETING_VERSION`); update `CHANGELOG.md` in both English and Chinese (separated by a divider) for releases.
